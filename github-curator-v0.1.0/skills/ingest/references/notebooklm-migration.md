# notebooklm-migration.md

The procedure for converting a NotebookLM source bundle into an
Agent Skills spec-conformant skill folder. Loaded on demand by
`skills/ingest/SKILL.md` only when the input shape matches a
NotebookLM bundle.

## What a NotebookLM bundle looks like

A NotebookLM project, when exported or replicated, has these signals:

- A list of "sources": web pages, PDFs, Google Docs, copy-pasted
  text. Each source has a title, an origin (URL or filename), and a
  body.
- One or more "notes": shorter pieces of writing the human created
  inside the notebook.
- An optional "audio overview" or "study guide" generated by
  NotebookLM. These are derivative; the skill ignores them during
  migration.

The export, when produced via tools like `stonega/notebooklm-tools`,
arrives as either:

- A zip containing one folder per source plus a `manifest.json`.
- A flat folder with files named after sources.

## Migration target

The migrated skill is one folder under `skills/<n>/` containing:

- `SKILL.md` with frontmatter (`name`, `description`, optional
  `license` and `compatibility`).
- `references/` containing one markdown file per source that the
  human classifies as reference material.
- `assets/` containing PDFs, images, and other binary attachments.

## The migration loop

```
INVENTORY -> CLASSIFY -> COMPOSE -> INTERROGATE -> WRITE
```

### INVENTORY

Walk the bundle. For each item, record:
- File path.
- Inferred type: source body, note, image, audio overview.
- Original URL or filename.
- Size.
- sha256.

Save the inventory to `/tmp/ingest/<timestamp>/notebooklm-inventory.json`.

### CLASSIFY

For each item, propose a target role:
- One source may become the SKILL.md body if it is the most
  comprehensive overview.
- Other sources become `references/<n>.md`.
- Notes become `references/notes-<topic>.md`.
- PDFs and images become `assets/<original-filename>`.
- Audio overviews are excluded.

The proposal is a write-list. It is not executed.

### COMPOSE

Draft proposed frontmatter:
- `name`: derived from the notebook's title, normalized to
  lowercase-hyphenated, ≤64 chars.
- `description`: drafted from the notebook's stated purpose if
  present, or from the largest source's first paragraph. The
  agent flags that the "when to load" clause needs human input.
- `license`: defaults to Apache-2.0 unless the human overrides.
  If any source has a non-Apache-compatible license (CC-BY-NC,
  proprietary), the migration halts and asks.
- `compatibility`: defaults to the conservative single host
  "Claude.ai projects".

Draft the SKILL.md body:
- The body should describe what the skill does and when to load it.
- The body must NOT inline the reference content. It points to
  references via the standard "Reference docs" section.

### INTERROGATE

Per the Socratic Method (max 3 load-bearing questions per turn),
ask the human:

1. "The notebook's title is `<title>`. Use `name: <slug>` where
   slug is the lowercase-hyphenated normalization?"
2. "The proposed `description` is N chars and covers what the
   skill does. It does not cover when the agent should load it.
   Add a 'Load this skill when ...' clause, or approve the
   current text?"
3. "Source #N (`<title>`) has license `<X>` per its metadata.
   Migrate under Apache-2.0 (re-licensable), exclude (license
   incompatible), or migrate as `references/<n>.md` with the
   original license preserved (mixed-license bundle)?"

The human's answers populate the proposal. If the human declines a
question, the corresponding field is recorded as a gap and the
migration proceeds with the gap noted in the QUARANTINE record (if
the gap is essential) or as a flag (if optional).

### WRITE

1. Create `skills/<slug>/`.
2. Write `SKILL.md` with confirmed frontmatter and body.
3. Write each classified source to its target path.
4. Append a row to `registry.yaml`.
5. Update `CHANGELOG.md` `[Unreleased]` with an Added entry.
6. Run the full publish-readiness rubric (the SCORE step in the
   parent ingest loop).

## License compatibility table

The migration's licensing logic:

| Source license | Compatible with Apache-2.0 target? | Action |
|---|---|---|
| Public domain | Yes | Migrate. |
| Apache-2.0 | Yes | Migrate, preserve original notice in source file. |
| MIT | Yes | Migrate, preserve original notice. |
| BSD (2/3-clause) | Yes | Migrate, preserve original notice. |
| CC-BY 4.0 | Yes (with attribution) | Migrate, preserve attribution. |
| CC-BY-SA 4.0 | Conditionally (downstream effects) | Ask. |
| CC-BY-NC 4.0 | No (NC vs commercial-permissive Apache-2.0) | Exclude or quarantine. |
| GPL (any version) | No (copyleft incompatible with Apache-2.0) | Exclude. |
| Proprietary / unspecified | No | Exclude until human secures permission. |

The agent does not assume "probably fine" for unclear licenses. The
default for unclear is exclude with a question to the human.

## Common pitfalls

- **Overstuffed SKILL.md**: A NotebookLM with 30 sources tempts the
  agent to inline summaries into the body. The body should stay
  under 500 lines. References absorb the volume.
- **Lost provenance**: Each migrated source should carry its origin
  URL or filename in a comment block at the top of its
  `references/<n>.md`. The bundle must remain auditable.
- **Renamed without trace**: If a source's original filename is
  changed during migration, the inventory.json record is the audit
  trail. Do not delete the inventory until the migration commits.
- **Audio overview leakage**: NotebookLM's generated audio is
  derivative content. The agent excludes it by default.

## When migration is the wrong move

Some NotebookLM bundles are reference libraries, not skills. If the
catalogued sources are 80%+ raw research material with no clear
"when to load" trigger, the right output is a documentation site or
a wiki, not a skill. The agent surfaces this verdict and asks the
human whether to (a) proceed with skill migration anyway,
(b) export to a wiki-style format instead, or (c) split into both.
